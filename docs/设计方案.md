# 工业测控代码生成软件设计方案

## 1. 文档概述

本文档依据《项目文档.md》中的产品需求与技术要求，并结合 `demo.json` 所体现的 DaoSCADA/HT 组态描述文件结构，提出工业测控代码生成软件的总体设计方案。系统目标是以 AI 能力赋能 DaoSCADA 平台，自动完成设备模型、组态界面、控件素材、数据绑点与控制脚本生成，降低人工组态成本，提升生成内容的规范性、可校验性和可落地性。

## 2. 建设目标

### 2.1 总体目标

构建一套可集成至 DaoSCADA 平台的 AI 代码生成与组态辅助系统，支持从自然语言、设备说明书、变量清单等输入中自动生成可被 DaoSCADA 渲染和运行的工程资产。

### 2.2 具体目标

- 自动解析设备说明书，生成 DaoSCADA 设备模型。
- 根据自然语言描述与控件库生成组态界面 JSON。
- 自动生成设备模型与 PLC 变量之间的绑点文件。
- 自动生成 DaoSCADA 支持的控制脚本代码。
- 提供实时预览、布局质检、局部微调、修改确认与撤销能力。
- 以插件或内嵌服务方式与 DaoSCADA 系统集成。

## 3. 设计依据

### 3.1 需求依据

系统需满足以下核心要求：

- 后端采用 FastAPI，支持高并发与跨平台部署。
- 前端采用 Vue 或 React，支持 Canvas/SVG 渲染、缩放、拖拽与预览。
- 素材库采用 SQLite + 文件系统，向量库采用 Chroma 本地部署。
- JSON Schema 遵循 Draft-07 标准，支持严格校验。
- AI 输出必须经过规则约束、Schema 校验与业务质检后方可使用。

### 3.2 `demo.json` 结构依据

`demo.json` 展示了 DaoSCADA 组态文件的典型结构，主要包括：

- `v`：组态文件版本号，例如 `8.0.5`。
- `p.layers`：图层配置，包含图层名称、可见性、可选择性、可移动性、可编辑性等。
- `p.autoAdjustIndex`、`p.hierarchicalRendering`：渲染与层级控制参数。
- `a`：画布交互属性，如 `fitContent`、`zoomable`、`pannable`。
- `d`：控件数据数组，每个元素描述一个 DaoSCADA/HT 节点。
- `contentRect`：画布内容边界，包含 `x`、`y`、`width`、`height`。

其中控件节点典型字段如下：

- `c`：控件类名，例如 `ht.Node`。
- `i`：控件唯一标识。
- `p.displayName`：控件显示名称。
- `p.image`：控件素材路径，例如 `symbols/_public/devices/高低温试验箱.json`。
- `p.position`：控件坐标。
- `p.width`、`p.height`：控件尺寸。
- `p.dataBindings.a`：属性级数据绑定配置。
- 绑定项包含 `key`、`text`、`tooltip`、`path` 等字段。

因此，本系统生成的组态 JSON 应兼容 DaoSCADA 现有渲染引擎，并在内部规则层将 PRD 中定义的 `canvas/control/binding` 概念映射到 DaoSCADA 实际文件结构。

## 4. 总体架构设计

### 4.1 架构分层

系统采用“前端交互层 + 后端服务层 + AI Agent 层 + 规则校验层 + 数据存储层 + DaoSCADA 集成层”的分层架构。

```text
用户/开发者
   │
   ▼
前端交互层：需求输入、素材管理、组态预览、微调、绑点复核
   │
   ▼
后端服务层：FastAPI 网关、任务编排、文件解析、接口聚合
   │
   ▼
AI Agent 层：设备建模、素材生成、自动布局、JSON Patch 微调、绑点匹配、脚本生成
   │
   ▼
规则校验层：JSON Schema、枚举字典、布局约束、绑点冲突检测、质检工具
   │
   ▼
数据存储层：SQLite、文件系统、Chroma 向量库、日志与版本记录
   │
   ▼
DaoSCADA 集成层：组态 JSON 导入、设备模型导入、绑点文件导入、脚本导入
```

### 4.2 技术选型

| 模块 | 技术方案 | 说明 |
| --- | --- | --- |
| 后端框架 | FastAPI | 提供 REST API、异步任务接口、Schema 校验服务 |
| 前端框架 | Vue 3 或 React | 实现组态预览、素材管理、绑点复核、AI 微调界面 |
| 渲染方式 | Canvas/SVG/HT 渲染适配 | 支持 DaoSCADA 组态 JSON 实时预览 |
| 本地数据库 | SQLite | 存储素材元数据、任务记录、历史版本、枚举字典索引 |
| 文件存储 | 文件系统 | 存储图标、控件 JSON、说明书、变量清单、导出文件 |
| 向量库 | Chroma | 存储素材、控件、变量、设备说明的 embedding |
| Schema 校验 | jsonschema | 按 Draft-07 校验组态 JSON 与绑点 JSON |
| 布局算法 | 规则布局 + D3-force/自研力导向算法 | 小场景规则布局，大场景辅助排布 |
| AI 能力 | LLM API、文生图 API、Embedding API | 支持自然语言理解、生成、检索增强与素材生成 |

### 4.3 基于 RAG 思路的模块划分

本项目核心技术路线是 RAG（Retrieval-Augmented Generation，检索增强生成）。系统不直接让大模型凭空生成设备模型、组态 JSON、绑点和脚本，而是先构建本地知识库，再围绕“资料入库 -> 检索召回 -> 上下文构造 -> 受控生成 -> 校验修复 -> 反馈回流”的链路划分模块。

#### 4.3.1 RAG 主链路

```text
知识采集
  -> 文档解析与结构化抽取
  -> 分块、标签化、向量化
  -> 知识库存储
  -> 任务理解与路由
  -> 多路检索与重排
  -> Prompt 上下文构造
  -> AI Agent 受控生成
  -> Schema / 业务 / 安全校验
  -> 预览复核与导出
  -> 用户反馈回流
```

#### 4.3.2 模块划分

| 模块 | 定位 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| 知识采集与解析模块 | 将说明书、点表、历史工程、素材、脚本模板转为可检索知识 | PDF/Word/Excel/JSON/图片/脚本 | 结构化文本、表格行、JSON 节点、素材元数据 |
| 知识分块与向量化模块 | 按 RAG 检索粒度生成 chunk 与 embedding | 解析后的文本、表格、控件、变量、脚本 | `knowledge_chunk`、embedding、metadata |
| 知识库存储模块 | 管理 SQLite、文件系统、Chroma 三类存储 | 原始文件、chunk、向量、元数据 | 可版本化、可追溯的本地知识库 |
| 检索与重排模块 | 按任务召回最相关的规范、设备、变量、素材和案例 | 用户 Query、任务类型、项目上下文 | Top-K 知识片段、候选变量、候选素材、相似案例 |
| Prompt 与上下文构造模块 | 将检索结果压缩为可控 Prompt | 检索结果、Schema、规则、任务参数 | 任务 Prompt、输出格式约束、引用证据 |
| Agent 生成模块 | 根据上下文生成设备模型、布局、绑点、Patch、脚本 | Prompt、检索证据、输出 Schema | 结构化 JSON、JSON Patch、脚本草案 |
| 规则校验与自动修复模块 | 对 AI 输出进行强约束检查和局部修复 | AI 输出、Schema、枚举、业务规则 | 校验报告、修复 Patch、可用结果 |
| 前端预览与人工复核模块 | 展示结果、差异、引用依据并收集人工确认 | 生成结果、校验报告、引用来源 | 确认结果、修改记录、复核意见 |
| DaoSCADA 导出集成模块 | 将内部结果转换为 DaoSCADA 可导入资产 | 设备模型、布局 JSON、绑点、脚本 | DaoSCADA 组态 JSON、绑点文件、脚本文件 |
| 反馈回流与审计模块 | 将确认结果沉淀为高质量案例并记录过程 | 用户反馈、最终导出文件、任务日志 | 历史案例、审计日志、优化样本 |

#### 4.3.3 模块协作关系

```text
前端交互层
  -> 知识采集与解析模块
  -> 知识分块与向量化模块
  -> 知识库存储模块
  -> 检索与重排模块
  -> Prompt 与上下文构造模块
  -> Agent 生成模块
  -> 规则校验与自动修复模块
  -> 前端预览与人工复核模块
  -> DaoSCADA 导出集成模块
  -> 反馈回流与审计模块
```

其中，知识采集、知识分块、知识库存储构成 RAG 的“知识准备侧”；检索重排、Prompt 构造、Agent 生成构成 RAG 的“生成执行侧”；规则校验、人工复核、导出集成、反馈回流构成 RAG 的“可靠落地侧”。

## 5. 核心模块具体实现

本章按照 4.3 的 RAG 模块划分，逐一说明每个模块的输入、处理逻辑、输出结果和关键实现点。

### 5.1 知识采集与解析模块

#### 5.1.1 模块目标

将 DaoSCADA 规范、`demo.json`、设备说明书、PLC 点表、素材文件、历史组态工程和脚本模板解析为后续可分块、可检索、可追溯的基础数据。

#### 5.1.2 输入范围

| 输入类型 | 示例 | 解析重点 |
| --- | --- | --- |
| 组态 JSON | `demo.json`、历史工程文件 | `v/p/a/d/contentRect`、控件节点、绑定路径、图层配置 |
| Schema 与枚举 | Draft-07 Schema、枚举字典 | 字段类型、必填项、枚举值、业务约束 |
| 设备说明书 | PDF、Word、文本 | 设备名称、参数、单位、状态、报警、控制说明 |
| PLC 点表 | Excel、CSV | 变量名、地址、数据类型、单位、读写权限、设备编号 |
| 素材文件 | 控件 JSON、图标、图片 | 素材名称、类型、尺寸、风格、路径、可绑定属性 |
| 脚本模板 | DaoSCADA 脚本样例 | 变量引用、控制动作、安全边界、注释规范 |

#### 5.1.3 实现流程

```text
上传文件
  -> 文件类型识别
  -> 文件安全检查
  -> 内容解析
  -> 结构化字段抽取
  -> 解析结果标准化
  -> 保存原始文件和解析结果
```

关键实现：

- PDF/Word 说明书优先抽取标题、段落、表格和页码，便于后续引用溯源。
- Excel 点表需支持表头模糊匹配，例如“变量名称”“点名”“TagName”统一映射为 `variable_name`。
- 组态 JSON 需按画布、图层、控件、绑定项拆解，保留每个控件的 `i`、`displayName`、`image`、`position`、`dataBindings`。
- 素材文件需生成文本化描述，例如“工业风高低温试验箱控件，默认尺寸 220×160，路径 symbols/...”。

### 5.2 知识分块与向量化模块

#### 5.2.1 模块目标

将解析后的资料切分为适合 RAG 检索的 chunk，并生成 embedding 写入 Chroma。

#### 5.2.2 分块策略

| 知识类型 | 分块粒度 | Metadata |
| --- | --- | --- |
| 设备说明书 | 章节、参数表、控制逻辑段落 | `device_type`、章节、页码、参数名 |
| PLC 点表 | 单变量行 + 设备变量集合 | 设备名、变量名、地址、单位、读写权限 |
| 组态 JSON | 画布、图层、控件节点、绑定项 | 控件 ID、控件名、素材路径、绑定路径 |
| 素材库 | 单个素材或控件 | 类型、风格、尺寸、标签、文件路径 |
| 布局案例 | 单个画面或区域 | 工艺类型、画布尺寸、设备数量、区域类型 |
| 脚本模板 | 单个控制场景 | 控制动作、变量权限、安全条件 |
| 规则知识 | 单条 Schema/枚举/业务规则 | 规则类型、适用模块、错误码 |

#### 5.2.3 输出结构

```json
{
  "chunkId": "chunk-001",
  "sourceId": "manual-001",
  "module": "device_model",
  "content": "温度测量值，单位 ℃，只读，表示箱体当前温度。",
  "metadata": {
    "device_type": "高低温试验箱",
    "content_type": "parameter",
    "page": 12,
    "tags": ["温度", "测量值"]
  }
}
```

关键实现：chunk 文本应包含足够上下文，但避免过长；Chroma 中保存向量和 metadata，SQLite 中保存 chunk 主记录，二者通过 `chunk_id` 关联。

### 5.3 知识库存储模块

#### 5.3.1 模块目标

建立 SQLite + 文件系统 + Chroma 的本地知识库，满足离线部署、版本管理和审计追溯要求。

#### 5.3.2 存储分工

| 存储 | 保存内容 | 作用 |
| --- | --- | --- |
| 文件系统 | 原始上传文件、素材、导出 JSON、脚本文件 | 保存大文件和最终交付物 |
| SQLite | 项目、素材元数据、设备模型、变量、chunk、任务、日志 | 结构化管理与版本追踪 |
| Chroma | chunk embedding、素材向量、变量向量、案例向量 | 相似度检索 |

#### 5.3.3 集合设计

- `asset_embeddings`：控件素材向量。
- `device_embeddings`：说明书、设备模型、设备参数向量。
- `variable_embeddings`：变量名、参数名、点位说明向量。
- `layout_case_embeddings`：历史组态布局案例向量。
- `script_embeddings`：脚本模板、安全规则、变量引用样例向量。
- `rule_embeddings`：Schema、枚举字典、业务质检规则向量。

关键实现：每条向量记录必须带 `project_id`、`source_id`、`chunk_id`、`version`，避免不同项目知识混用。

### 5.4 检索与重排模块

#### 5.4.1 模块目标

根据用户任务检索最相关的规范、设备、变量、素材、案例和脚本模板，为生成提供证据。

#### 5.4.2 检索流程

```text
接收任务请求
  -> Query 改写
  -> 任务类型识别
  -> 多集合召回
  -> 元数据过滤
  -> 规则加权重排
  -> Top-K 上下文输出
```

#### 5.4.3 不同任务的检索组合

| 任务 | 检索集合 | 重排因子 |
| --- | --- | --- |
| 设备模型生成 | `device_embeddings`、`variable_embeddings` | 设备类型、参数名、单位、读写属性 |
| 素材检索/生成 | `asset_embeddings`、`device_embeddings` | 素材类型、风格、尺寸、相似度阈值 |
| 自动布局 | `layout_case_embeddings`、`asset_embeddings`、`device_embeddings`、`rule_embeddings` | 工艺类型、设备数量、画布尺寸 |
| 微调 | 当前 JSON、`layout_case_embeddings`、`rule_embeddings` | 选中控件、操作意图、Patch 安全规则 |
| 数据绑点 | `variable_embeddings`、`device_embeddings` | 语义相似度、设备编号、单位、读写权限 |
| 控制脚本 | `script_embeddings`、`variable_embeddings`、`rule_embeddings` | 控制动作、写点权限、安全边界 |

关键实现：检索不能只依赖 embedding，相同编号、单位一致、读写权限匹配等规则应参与重排，尤其是数据绑点和脚本生成。

### 5.5 Prompt 与上下文构造模块

#### 5.5.1 模块目标

将检索结果压缩、排序并组装为面向不同 Agent 的受控 Prompt，保证 AI 只能基于证据生成。

#### 5.5.2 Prompt 组成

```text
系统角色：DaoSCADA 工业组态生成助手
任务说明：当前任务类型、目标、输出范围
用户需求：自然语言 Query 和补充参数
检索证据：规范、设备、变量、素材、案例、脚本模板
约束规则：Schema、枚举、布局规则、权限规则、安全规则
输出格式：JSON / JSON Patch / 脚本模板
禁止事项：不得编造字段、路径、变量地址、素材路径
```

#### 5.5.3 输出约束

- 设备模型生成只允许输出设备模型 JSON。
- 布局生成只允许输出 DaoSCADA 兼容 JSON 或内部布局 DSL。
- 微调只允许输出 RFC 6902 JSON Patch。
- 绑点只允许输出候选列表和确认后的 `dataBindings`。
- 脚本生成必须附带变量说明和安全前置条件。

### 5.6 Agent 生成模块

#### 5.6.1 模块目标

在 RAG 上下文约束下完成设备模型、素材、布局、微调、绑点和脚本生成。

#### 5.6.2 Agent 划分

| Agent | 输入 | 输出 | 实现重点 |
| --- | --- | --- | --- |
| 设备建模 Agent | 说明书片段、点表、同类设备模型 | 设备模型 JSON | 参数分类、单位、读写属性、状态枚举 |
| 素材 Agent | 素材候选、设备类型、风格约束 | 素材复用结果或生成提示词 | 优先复用，相似度不足再生成 |
| 布局 Agent | 设备模型、素材、布局案例、规则 | 区域规划、控件清单、DaoSCADA JSON | AI 负责规划，坐标优先由规则算法生成 |
| 微调 Agent | 当前 JSON、选中控件、修改指令 | JSON Patch | 限制 Patch 路径和操作范围 |
| 绑点 Agent | 控件绑定需求、变量候选、规则 | Top-N 候选、绑定结果 | AI 推荐，人工确认 |
| 脚本 Agent | 控制逻辑、变量权限、脚本模板 | 脚本草案 | 安全边界和写点权限优先 |

#### 5.6.3 典型生成闭环

```text
任务输入
  -> 检索上下文
  -> Prompt 构造
  -> Agent 生成
  -> 结构化解析
  -> 校验
  -> 必要时局部修复
  -> 返回结果与引用依据
```

### 5.7 规则校验与自动修复模块

#### 5.7.1 模块目标

保证 AI 输出可被 DaoSCADA 正确解析，并满足工业现场安全要求。

#### 5.7.2 校验类型

| 校验类型 | 内容 |
| --- | --- |
| Schema 校验 | Draft-07，检查字段完整性、类型、数组、对象结构 |
| 枚举校验 | 控件类名、图层属性、绑定类型、素材类型必须来自字典 |
| 布局校验 | 越界、重叠、孤立管路、重复 ID、尺寸异常 |
| 绑点校验 | 地址冲突、变量重复、单位不一致、读写权限不匹配 |
| 脚本安全校验 | 强制写入、无限循环写入、缺少上下限、缺少互锁条件 |

#### 5.7.3 自动修复策略

可自动修复的问题包括字段缺省、坐标轻微越界、控件重叠、Patch 路径格式错误等；涉及变量地址、写点权限、安全边界的问题必须转人工确认。

校验报告示例：

```json
{
  "valid": false,
  "errors": [
    {
      "path": "d[3].p.position.x",
      "type": "layout.out_of_bounds",
      "message": "控件超出 contentRect 边界",
      "suggestion": "将 x 调整到 0 至 1920 范围内"
    }
  ],
  "repairable": true
}
```

### 5.8 前端预览与人工复核模块

#### 5.8.1 模块目标

让用户在导出前看到生成结果、差异、引用依据和质量报告，并对关键结果进行确认。

#### 5.8.2 页面实现

- **知识库管理页**：查看文件解析、分块、向量化状态。
- **组态预览页**：渲染 DaoSCADA JSON，支持缩放、拖拽、控件选择、局部微调。
- **绑点复核页**：展示 Top-N 候选、置信度、匹配依据，支持人工改绑。
- **差异确认页**：展示 JSON Patch 前后差异，支持接受和撤销。
- **生成溯源页**：展示引用的说明书章节、点表行、素材、历史案例和规则版本。

#### 5.8.3 人工复核原则

- 低置信度绑点不得自动写入正式文件。
- 控制脚本涉及写点时必须人工确认。
- 自动修复后的布局需重新预览。
- 用户确认后的结果才能进入高质量案例库。

### 5.9 DaoSCADA 导出集成模块

#### 5.9.1 模块目标

将内部生成结果转换为 DaoSCADA 可导入、可回滚、可审计的工程资产。

#### 5.9.2 导出内容

| 导出类型 | 输出内容 |
| --- | --- |
| 组态 JSON | `v/p/a/d/contentRect`，兼容 DaoSCADA 渲染引擎 |
| 设备模型 | 设备基础信息、参数、状态量、测量量、设定量、控制量 |
| 绑点文件 | `p.dataBindings.a` 绑定项、变量路径、tooltip、冲突报告 |
| 控制脚本 | 脚本文件、变量说明、安全检查报告 |

#### 5.9.3 兼容映射

```text
canvas   -> v / p / a / contentRect
control  -> d[].c / d[].i / d[].p.displayName / d[].p.image / d[].p.position / width / height
binding  -> d[].p.dataBindings.a.*.key / text / tooltip / path
```

导出前必须通过 Schema、业务规则和人工确认；导出后记录版本快照，支持回滚。

### 5.10 反馈回流与审计模块

#### 5.10.1 模块目标

将用户确认结果沉淀为高质量知识，并完整记录每次 RAG 生成过程。

#### 5.10.2 反馈回流流程

```text
用户确认结果
  -> 标记为高质量样例
  -> 生成摘要和标签
  -> 写入历史案例库
  -> 更新 Chroma embedding
  -> 后续相似任务优先召回
```

#### 5.10.3 审计记录

每次任务需记录：用户输入摘要、检索到的 chunk、Prompt 摘要、AI 输出摘要、校验报告、修复记录、用户确认结果、最终导出文件版本。

### 5.11 MVP 实施优先级

| 优先级 | 模块 | MVP 范围 |
| --- | --- | --- |
| P0 | 知识采集与解析、知识分块与向量化、知识库存储 | 支持 `demo.json`、Schema、Excel 点表、说明书文本入库 |
| P0 | 检索与重排、规则校验 | 支持按任务类型检索，组态 JSON 和绑点 JSON 可校验 |

## 8. 关键流程设计

### 8.1 组态界面生成流程

```text
用户输入自然语言需求

  -> 布局质检
```

### 8.2 数据绑点流程

```text
上传 Excel 点表
  -> 表头识别与变量解析
  -> 变量标准化与向量化
  -> 提取组态控件绑定需求
  -> 语义匹配与规则过滤

  -> 冲突检测
  -> 人工复核
  -> 写入 dataBindings
  -> 导出绑点 JSON
```

```text
前端选择控件

  -> 前端差异预览
  -> 接受/撤销
  -> 更新完整组态 JSON
```

## 9. 质量保障设计

### 9.1 校验机制

- 所有 AI 输出必须经过 JSON Schema 校验。
- 所有枚举字段必须来自枚举字典。
- 所有布局结果必须经过重叠、越界、孤立控件检测。
- 所有绑点结果必须经过地址冲突、变量重复、读写权限检测。
- 所有脚本必须经过静态检查与安全规则检查。

### 9.2 版本管理

- Schema、枚举字典、设备模型、组态 JSON、绑点文件均需记录版本。
- 每次 AI 生成与用户修改应保存历史快照。
- 支持回滚至任意已确认版本。

### 9.3 日志与审计

- 记录 AI 请求、输入摘要、输出摘要、校验结果。
- 记录用户接受、撤销、手动修改等操作。
- 对第三方 API 调用记录耗时、状态码、失败原因。

## 10. 安全与可靠性设计

- 支持离线部署，本地存储素材、Schema、点表、设备模型与向量索引。
- 对上传文件进行类型、大小、内容安全检查。
- 第三方 API 密钥采用环境变量或密钥管理服务保存，不写入代码仓库。
- 控制脚本生成必须限制危险操作，禁止生成未确认的强制写入逻辑。
- 关键导出文件生成前必须经过校验与人工确认。
- 数据库和文件系统定期备份，防止素材与工程数据丢失。

## 11. 性能设计

| 指标 | 设计方案 |
| --- | --- |
| 后端接口响应 | 常规接口目标 ≤500ms，不含第三方 API 耗时 |
| 预览渲染 | 组态 JSON 解析与首屏渲染目标 ≤1000ms |
| 素材管理 | 浏览、搜索、删除操作目标 ≤100ms |
| 批量素材生成 | 目标 ≥10 个/分钟 |
| 布局大场景 | 超过 20 元素启用算法辅助布局 |
| 向量检索 | 本地 Chroma 检索，减少外部依赖 |

## 13. 验收设计

### 13.1 Schema 与规则验收

- 提供不少于 3 个合法 JSON 示例和 3 个非法 JSON 示例。
- 校验工具对合法/非法 JSON 判断准确率达到 100%。
- 字段缺失、类型错误、枚举越界均能给出准确错误位置与建议。

### 13.2 素材生成验收

- 5 条不同 Query 检索准确率不低于 90%。
- 新生成素材能成功入库并在管理界面展示。
- 批量生成效率不低于 10 个/分钟，合格率不低于 95%。

### 13.3 布局生成验收

- 至少 3 个典型场景布局 JSON 通过 Schema 校验。
- 前端预览正常渲染，无明显控件重叠、管路连接错误。
- 布局逻辑经人工评审合格。

### 13.4 微调能力验收

- 5 条典型调整指令响应准确率不低于 80%。
- JSON Patch 可正确应用。
- 撤销功能可完全回滚。

### 13.5 数据绑点验收

- 标准清单自动匹配 Top-1 准确率不低于 75%。
- 能识别预置寄存器冲突、变量重复等场景。
- 生成的绑点 JSON 通过 Schema 校验。

## 15. 风险与应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| AI 生成 JSON 不稳定 | 导致无法渲染或字段错误 | 使用 Schema、枚举字典、业务规则多层校验 |
| 素材风格不一致 | 影响界面美观与可用性 | 建立素材模板、风格检测与人工审核流程 |
| 变量匹配准确率不足 | 绑点错误影响运行 | 引入人工复核、同义词字典、编号归一化规则 |
| 第三方 API 不稳定 | 影响生成效率 | 建立重试、降级、本地缓存和离线模板能力 |
| DaoSCADA 兼容性差异 | 导入失败或渲染异常 | 以 `demo.json` 为基准建立兼容映射与回归测试集 |

## 16. 结论

本方案通过规则约束、AI Agent 编排、本地素材与向量检索、DaoSCADA JSON 兼容映射、前端实时预览和人工复核机制，形成一套可落地的工业测控代码生成软件设计。系统既利用 AI 提升组态、绑点和脚本生成效率，又通过 Schema 校验、布局质检、冲突检测和版本回滚保障工业场景下的可靠性与安全性。
