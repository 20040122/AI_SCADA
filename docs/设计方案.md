# 工业测控代码生成软件设计方案

## 1. 文档概述

本文档依据《项目文档.md》中的产品需求与技术要求，并结合 `demo.json` 所体现的 DaoSCADA/HT 组态描述文件结构，提出工业测控代码生成软件的总体设计方案。系统目标是以 AI 能力赋能 DaoSCADA 平台，自动完成设备模型、组态界面、控件素材、数据绑点与控制脚本生成，降低人工组态成本，提升生成内容的规范性、可校验性和可落地性。

## 2. 建设目标

### 2.1 总体目标

构建一套可集成至 DaoSCADA 平台的 AI 代码生成与组态辅助系统，支持从自然语言、设备说明书、变量清单等输入中自动生成可被 DaoSCADA 渲染和运行的工程资产。

### 2.2 具体目标

- 自动解析设备说明书，生成 DaoSCADA 设备模型。
- 根据自然语言描述与控件库生成组态界面 JSON。
- 自动生成设备模型与 PLC 变量之间的绑点文件。
- 自动生成 DaoSCADA 支持的控制脚本代码。
- 提供实时预览、布局质检、局部微调、修改确认与撤销能力。
- 以插件或内嵌服务方式与 DaoSCADA 系统集成。

## 3. 设计依据

### 3.1 需求依据

系统需满足以下核心要求：

- 后端采用 FastAPI，支持高并发与跨平台部署。
- 前端采用 Vue 或 React，支持 Canvas/SVG 渲染、缩放、拖拽与预览。
- 素材库采用 SQLite + 文件系统，向量库采用 Chroma 本地部署。
- JSON Schema 遵循 Draft-07 标准，支持严格校验。
- AI 输出必须经过规则约束、Schema 校验与业务质检后方可使用。

### 3.2 `demo.json` 结构依据

`demo.json` 展示了 DaoSCADA 组态文件的典型结构，主要包括：

- `v`：组态文件版本号，例如 `8.0.5`。
- `p.layers`：图层配置，包含图层名称、可见性、可选择性、可移动性、可编辑性等。
- `p.autoAdjustIndex`、`p.hierarchicalRendering`：渲染与层级控制参数。
- `a`：画布交互属性，如 `fitContent`、`zoomable`、`pannable`。
- `d`：控件数据数组，每个元素描述一个 DaoSCADA/HT 节点。
- `contentRect`：画布内容边界，包含 `x`、`y`、`width`、`height`。

其中控件节点典型字段如下：

- `c`：控件类名，例如 `ht.Node`。
- `i`：控件唯一标识。
- `p.displayName`：控件显示名称。
- `p.image`：控件素材路径，例如 `symbols/_public/devices/高低温试验箱.json`。
- `p.position`：控件坐标。
- `p.width`、`p.height`：控件尺寸。
- `p.dataBindings.a`：属性级数据绑定配置。
- 绑定项包含 `key`、`text`、`tooltip`、`path` 等字段。

因此，本系统生成的组态 JSON 应兼容 DaoSCADA 现有渲染引擎，并在内部规则层将 PRD 中定义的 `canvas/control/binding` 概念映射到 DaoSCADA 实际文件结构。

## 4. 总体架构设计

### 4.1 架构分层

系统采用“前端交互层 + 后端服务层 + AI Agent 层 + 规则校验层 + 数据存储层 + DaoSCADA 集成层”的分层架构。

```text
用户/开发者
   │
   ▼
前端交互层：需求输入、素材管理、组态预览、微调、绑点复核
   │
   ▼
后端服务层：FastAPI 网关、任务编排、文件解析、接口聚合
   │
   ▼
AI Agent 层：设备建模、素材生成、自动布局、JSON Patch 微调、绑点匹配、脚本生成
   │
   ▼
规则校验层：JSON Schema、枚举字典、布局约束、绑点冲突检测、质检工具
   │
   ▼
数据存储层：SQLite、文件系统、Chroma 向量库、日志与版本记录
   │
   ▼
DaoSCADA 集成层：组态 JSON 导入、设备模型导入、绑点文件导入、脚本导入
```

### 4.2 技术选型

| 模块 | 技术方案 | 说明 |
| --- | --- | --- |
| 后端框架 | FastAPI | 提供 REST API、异步任务接口、Schema 校验服务 |
| 前端框架 | Vue 3 或 React | 实现组态预览、素材管理、绑点复核、AI 微调界面 |
| 渲染方式 | Canvas/SVG/HT 渲染适配 | 支持 DaoSCADA 组态 JSON 实时预览 |
| 本地数据库 | SQLite | 存储素材元数据、任务记录、历史版本、枚举字典索引 |
| 文件存储 | 文件系统 | 存储图标、控件 JSON、说明书、变量清单、导出文件 |
| 向量库 | Chroma | 存储素材、控件、变量、设备说明的 embedding |
| Schema 校验 | jsonschema | 按 Draft-07 校验组态 JSON 与绑点 JSON |
| 布局算法 | 规则布局 + D3-force/自研力导向算法 | 小场景规则布局，大场景辅助排布 |
| AI 能力 | LLM API、文生图 API、Embedding API | 支持自然语言理解、生成、检索增强与素材生成 |

## 5. 核心模块设计

### 5.1 JSON Schema 规则库模块

#### 5.1.1 功能说明

建立组态描述文件规则库，对 AI 生成内容进行强约束，确保输出可被 DaoSCADA 渲染引擎直接解析。

#### 5.1.2 规则模型

内部规则按三类核心对象建模：

- `canvas`：画布信息，对应 `demo.json` 中的 `a`、`contentRect`、部分 `p` 配置。
- `control`：控件信息，对应 `d[]` 中的每一个 `ht.Node`。
- `binding`：绑点信息，对应 `p.dataBindings.a` 下的绑定项。

导出时映射为 DaoSCADA 实际结构：

```text
canvas   -> v / p / a / contentRect
control  -> d[].c / d[].i / d[].p.displayName / d[].p.image / d[].p.position / width / height
binding  -> d[].p.dataBindings.a.*.key / text / tooltip / path
```

#### 5.1.3 校验能力

- 字段完整性校验：检查版本、图层、画布、控件、坐标、绑定字段是否缺失。
- 类型校验：检查坐标、尺寸、标识、布尔值、数组、对象类型。
- 枚举校验：控件类名、图层属性、绑定类型、素材路径类型等使用枚举字典约束。
- 业务校验：检查控件重叠、无效坐标、孤立管路、绑定冲突等。
- 错误输出：返回错误行、错误路径、错误类型、修改建议。

### 5.2 AI 素材/控件生成模块

#### 5.2.1 输入

- 自然语言 Query，例如“水泵运行状态指示灯”。
- 设备类型列表。
- 目标风格、尺寸、颜色、工业场景等约束。

#### 5.2.2 处理流程

1. 对 Query 进行意图识别与实体抽取，识别设备类型、状态、用途、风格。
2. 调用 embedding 模型生成向量。
3. 在 Chroma 中检索本地素材库。
4. 相似度大于 0.85 时复用已有素材。
5. 无匹配素材时调用文生图 API 或控件模板生成器。
6. 对新素材进行尺寸、格式、透明背景、风格一致性质检。
7. 质检通过后写入文件系统，并将元数据写入 SQLite 与 Chroma。

#### 5.2.3 素材元数据

素材元数据建议包括：

- `id`：素材唯一标识。
- `name`：素材名称。
- `type`：设备、仪表、按钮、管路、状态灯等。
- `style`：工业风、扁平风、仿真风等。
- `path`：素材文件路径，对应 DaoSCADA 的 `p.image`。
- `width`、`height`：默认尺寸。
- `tags`：检索标签。
- `embedding_id`：向量库索引。
- `created_at`、`updated_at`：创建与更新时间。

### 5.3 AI 自动布局模块

#### 5.3.1 输入

- 用户自然语言描述。
- 画布宽高。
- 控件/素材列表。
- 设备关系、工艺流程、管路连接关系。

#### 5.3.2 布局策略

- 先由 LLM 将用户描述转换为 Mermaid 或流程图 DSL，形成控制流骨架。
- 根据骨架划分主设备区、控制区、数据显示区、报警区、管路区。
- 基于规则生成控件坐标、尺寸、层级与连接关系。
- 元素数量不超过 20 个时优先使用规则布局。
- 元素数量超过 20 个时采用力导向图算法辅助排布，再由 AI 微调。

#### 5.3.3 约束规则

- 同类设备距离小于 200px，并尽量聚合展示。
- 管路控件必须连接至少两个设备或管路节点。
- 仪表、数据显示、设定值输入控件默认位于右侧或下方控制面板。
- 主设备位于画布视觉中心。
- 控件不得超出 `contentRect` 边界。
- 控件重叠率超过 10% 时触发布局告警。

#### 5.3.4 输出

输出兼容 DaoSCADA 的组态 JSON。例如 `demo.json` 中“高低温试验箱”节点可由设备控件生成，其温度、湿度、运行状态等绑定项写入 `p.dataBindings.a`。

### 5.4 微调接口模块

#### 5.4.1 功能说明

在组态预览页面中，用户可选择单个或多个控件，通过自然语言指令对局部内容进行调整。

#### 5.4.2 交互流程

1. 用户点击或框选控件。
2. 前端提取选中控件的 `i`、`displayName`、`position`、`width`、`height`、`dataBindings` 等上下文。
3. 前端将选中控件上下文和完整组态 JSON 发送给后端。
4. AI 根据指令仅生成 RFC 6902 JSON Patch。
5. 后端校验 Patch 合法性与安全性。
6. 前端应用 Patch 并展示差异预览。
7. 用户点击“接受修改”或“撤销”。

#### 5.4.3 支持指令

- 换位：交换两个控件坐标。
- 对齐：左对齐、右对齐、水平居中、垂直居中。
- 缩放：放大、缩小、统一尺寸。
- 改色：修改控件样式属性或替换素材。
- 删除：移除选中控件及其绑定。
- 调整绑定：修改绑定路径或绑定参数。

### 5.5 数据绑点模块

#### 5.5.1 输入

- Excel 设备变量清单。
- 组态 JSON 中的控件列表与绑定候选项。
- 设备模型与变量字典。

#### 5.5.2 解析能力

系统需对 Excel 表头进行模糊匹配，提取：

- 变量名。
- 数据类型。
- 寄存器地址。
- 单位。
- 设备名称。
- 参数名称。
- 读写属性。

#### 5.5.3 匹配算法

- 基于控件 `displayName`、绑定属性名、变量名、设备名计算语义相似度。
- 结合编号归一化规则，例如“水泵1”“Pump_01”“PUMP-1”视为同一编号。
- 结合单位与数据类型进行二次过滤。
- 输出 Top-N 候选列表，并记录匹配依据。

#### 5.5.4 绑定输出结构

绑定结果应写入 DaoSCADA 控件属性中，例如：

```json
{
  "key": "direct-367564",
  "text": "温度测量值 (℃) [高低温箱#1]",
  "tooltip": "项目: fw的测试项目<br>区域: 组态测试<br>设备: 高低温箱#1<br>参数: 温度测量值 (℃)",
  "path": "direct,937,13805,367564"
}
```

其中 `key` 用于唯一标识绑定点，`text` 用于界面展示，`tooltip` 用于提示说明，`path` 用于 DaoSCADA 实际数据路径。

### 5.6 设备模型生成模块

#### 5.6.1 输入

- 设备说明书。
- 点表文件。
- 用户自然语言补充说明。

#### 5.6.2 输出

- 设备基础信息。
- 设备参数模型。
- 状态量、测量量、设定量、控制量分类。
- 默认控件推荐。
- 默认绑点候选。
- DaoSCADA 可导入的设备模型文件。

以 `demo.json` 中“高低温试验箱”为例，设备模型应至少抽取：

- 试验类型：`status.testType`
- 湿度测量值：`humidity.val`
- 温度测量值：`temperature.val`
- 温度设定值：`temperature.set`
- 湿度设定值：`humidity.set`
- 运行状态：`status.run`

### 5.7 控制脚本生成模块

#### 5.7.1 功能说明

根据设备模型、控制逻辑描述与 DaoSCADA 脚本规范，生成可执行或可导入的控制脚本。

#### 5.7.2 设计原则

- 脚本生成前需确认设备点位和读写权限。
- 涉及控制动作时必须加入安全边界检查。
- 输出脚本需附带注释与变量说明。
- 脚本需支持静态检查与模拟运行。

## 8. 关键流程设计

### 8.1 组态界面生成流程

```text
用户输入自然语言需求
  -> 需求解析与实体抽取
  -> 检索设备模型与控件素材
  -> 生成流程骨架/区域规划
  -> 自动布局生成控件坐标
  -> 生成 DaoSCADA JSON
  -> Schema 校验
  -> 布局质检
  -> 前端预览
  -> 用户确认或微调
  -> 导出组态文件
```

### 8.2 数据绑点流程

```text
上传 Excel 点表
  -> 表头识别与变量解析
  -> 变量标准化与向量化
  -> 提取组态控件绑定需求
  -> 语义匹配与规则过滤
  -> 输出 Top-N 候选
  -> 冲突检测
  -> 人工复核
  -> 写入 dataBindings
  -> 导出绑点 JSON
```

### 8.3 微调流程

```text
前端选择控件
  -> 提取选中控件上下文
  -> 输入自然语言调整指令
  -> AI 生成 JSON Patch
  -> Patch 合法性校验
  -> 前端差异预览
  -> 接受/撤销
  -> 更新完整组态 JSON
```

## 9. 质量保障设计

### 9.1 校验机制

- 所有 AI 输出必须经过 JSON Schema 校验。
- 所有枚举字段必须来自枚举字典。
- 所有布局结果必须经过重叠、越界、孤立控件检测。
- 所有绑点结果必须经过地址冲突、变量重复、读写权限检测。
- 所有脚本必须经过静态检查与安全规则检查。

### 9.2 版本管理

- Schema、枚举字典、设备模型、组态 JSON、绑点文件均需记录版本。
- 每次 AI 生成与用户修改应保存历史快照。
- 支持回滚至任意已确认版本。

### 9.3 日志与审计

- 记录 AI 请求、输入摘要、输出摘要、校验结果。
- 记录用户接受、撤销、手动修改等操作。
- 对第三方 API 调用记录耗时、状态码、失败原因。

## 10. 安全与可靠性设计

- 支持离线部署，本地存储素材、Schema、点表、设备模型与向量索引。
- 对上传文件进行类型、大小、内容安全检查。
- 第三方 API 密钥采用环境变量或密钥管理服务保存，不写入代码仓库。
- 控制脚本生成必须限制危险操作，禁止生成未确认的强制写入逻辑。
- 关键导出文件生成前必须经过校验与人工确认。
- 数据库和文件系统定期备份，防止素材与工程数据丢失。

## 11. 性能设计

| 指标 | 设计方案 |
| --- | --- |
| 后端接口响应 | 常规接口目标 ≤500ms，不含第三方 API 耗时 |
| 预览渲染 | 组态 JSON 解析与首屏渲染目标 ≤1000ms |
| 素材管理 | 浏览、搜索、删除操作目标 ≤100ms |
| 批量素材生成 | 目标 ≥10 个/分钟 |
| 布局大场景 | 超过 20 元素启用算法辅助布局 |
| 向量检索 | 本地 Chroma 检索，减少外部依赖 |

## 12. 部署设计

### 12.1 部署形态

- 单机离线部署：适用于内网工业现场。
- 局域网服务部署：后端服务统一部署，多个客户端访问。

## 13. 验收设计

### 13.1 Schema 与规则验收

- 提供不少于 3 个合法 JSON 示例和 3 个非法 JSON 示例。
- 校验工具对合法/非法 JSON 判断准确率达到 100%。
- 字段缺失、类型错误、枚举越界均能给出准确错误位置与建议。

### 13.2 素材生成验收

- 5 条不同 Query 检索准确率不低于 90%。
- 新生成素材能成功入库并在管理界面展示。
- 批量生成效率不低于 10 个/分钟，合格率不低于 95%。

### 13.3 布局生成验收

- 至少 3 个典型场景布局 JSON 通过 Schema 校验。
- 前端预览正常渲染，无明显控件重叠、管路连接错误。
- 布局逻辑经人工评审合格。

### 13.4 微调能力验收

- 5 条典型调整指令响应准确率不低于 80%。
- JSON Patch 可正确应用。
- 撤销功能可完全回滚。

### 13.5 数据绑点验收

- 标准清单自动匹配 Top-1 准确率不低于 75%。
- 能识别预置寄存器冲突、变量重复等场景。
- 生成的绑点 JSON 通过 Schema 校验。

## 15. 风险与应对

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| AI 生成 JSON 不稳定 | 导致无法渲染或字段错误 | 使用 Schema、枚举字典、业务规则多层校验 |
| 素材风格不一致 | 影响界面美观与可用性 | 建立素材模板、风格检测与人工审核流程 |
| 变量匹配准确率不足 | 绑点错误影响运行 | 引入人工复核、同义词字典、编号归一化规则 |
| 第三方 API 不稳定 | 影响生成效率 | 建立重试、降级、本地缓存和离线模板能力 |
| DaoSCADA 兼容性差异 | 导入失败或渲染异常 | 以 `demo.json` 为基准建立兼容映射与回归测试集 |

## 16. 结论

本方案通过规则约束、AI Agent 编排、本地素材与向量检索、DaoSCADA JSON 兼容映射、前端实时预览和人工复核机制，形成一套可落地的工业测控代码生成软件设计。系统既利用 AI 提升组态、绑点和脚本生成效率，又通过 Schema 校验、布局质检、冲突检测和版本回滚保障工业场景下的可靠性与安全性。
